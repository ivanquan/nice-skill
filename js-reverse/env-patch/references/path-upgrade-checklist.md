# Path Upgrade Checklist

用来判断当前补环境任务应该继续收敛，还是升级到 `vm`、高级 DOM/对象契约补丁或 `WASM`。

## 先别急着升级

任意一条成立时，通常还应该留在当前路径里收敛：

1. `ERRORS` 还没清零。
2. `undefinedPaths` / `UNDEFINED` 里还有明显高频缺项没确认。
3. 入口函数、调用参数、返回格式还没确认。
4. 请求失败更像 cookie/header/时序问题，而不是环境本身。
5. 还没做浏览器对照，无法确认偏差发生在哪一层。

## 升级到 vm

适合信号：

1. 代码动态下发，`eval`、`new Function`、首包拼接明显。
2. challenge 文档、外链 JS、动态参数需要运行期拿到后再执行。
3. 首跳 cookie 与后续动态 cookie 的更新关系依赖动态执行链。
4. 问题已确认在“执行载体”，不是普通宿主对象缺失。

不适合：

1. 只是少几个 `window/document/navigator` 字段。
2. 入口函数已经能单独拿到。
3. 当前样本未固定，仍在混用不同轮次动态脚本。

## 升级到高级 DOM / 对象契约补丁

适合信号：

1. 多轮补齐后仍反复卡在 DOM 结构、节点关系、布局值。
2. 目标显式依赖 `documentElement`、`body`、节点树、事件模型。
3. 存在 `toStringTag`、`instanceof`、构造器外形检查。
4. 不是单个缺项，而是整组浏览器行为被连续消费。

升级前先确认：

1. 最小环境确实走不通。
2. 已知道最先出偏差的是哪个宿主或方法。
3. 准备按致命级、高危级、中危级分层补，而不是一次全开。

## 升级到 WASM

适合信号：

1. 关键函数最终落到 `WebAssembly.instantiate` 或 `instantiateStreaming`。
2. JS 只负责桥接、拼参数、搬运内存。
3. 中间值在 JS 里看不全，最终结果主要由 `.wasm` 导出函数决定。

升级前先确认：

1. `.wasm` 文件已拿到。
2. `imports` 和 `exports` 已看过。
3. JS 侧喂给 wasm 的参数格式已明确。

## 简化决策

1. 还有明显缺项：留在最小环境。
2. 动态执行载入明显：升 `vm`。
3. 整套 DOM 行为参与：升高级 DOM / 对象契约补丁。
4. 核心逻辑在 `.wasm`：升 `WASM`。

升级不是重开一局，要保留入口定位结果、最小可验证事实和已确认不用补的结论。
