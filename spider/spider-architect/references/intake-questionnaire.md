# 需求与基础设施问卷（Intake Questionnaire）

Phase 0 与 Phase 0.5 使用本文件。**缺项必追问，不得假设默认值写入代码。**

---

## 使用规则（V1.4：逐 Checkpoint 停步）

1. 用户首次给出项目描述后，**只执行 Phase 0**：提问 + 摘要，**不写代码、不创建目录、不输出 Phase 1 及以后内容**。
2. 用户提供了「预设回答」时，视为部分答案；仍须核对缺项并追问；**不得因信息多而合并输出反爬/架构方案**。
3. **单轮单 Phase**：每完成一个 Phase，输出该 Phase 摘要 + 一个 CHECKPOINT，**立即停步**；用户明确确认后才进入下一 Phase。详见 [checkpoint-protocol.md](checkpoint-protocol.md)。
4. 敏感信息（API Key、密码）请用户自行填入 `config/settings.py` 或 `.env`；**禁止写入 skill 文件或 commit**。
5. 用户在某 CHECKPOINT 修改已确认项时，从**被改 Phase 的下一 Phase 起重新执行**，不沿用旧方案。

## Phase 0：业务需求诊断（4 必问 + 补充）

### 4 个必问

| # | 问题 | 影响 |
|---|------|------|
| Q1 | **数据规模与频次**：单次几万条，还是每天持续百万级？ | 选型、并发、存储 |
| Q2 | **并发与分布式**：单机还是多机？预期 QPS/线程数？ | Batch/Task、Redis |
| Q3 | **断点续传**：中断后是否必须从断点继续？ | BatchSpider vs AirSpider |
| Q4 | **调度复杂度**：简单定时 / 优先级 / 任务依赖 / 动态下发？ | TaskSpider vs 外部 Cron |

### 补充收集（按需追问）

| # | 问题 |
|---|------|
| Q5 | 目标网站 URL、数据类型（HTML/API/两者）、字段清单 |
| Q6 | 是否已有抓包/HAR/样本响应/账号 Cookie |
| Q7 | 技术约束（语言、框架、必须复用的内部工具） |
| Q8 | 项目名称、工作区落地路径 |
| Q9 | 数据用途声明（合规标注用，不阻断） |

### Phase 0 输出

- `feapder_mode` 建议（air / batch / task）+ 理由
- 诊断摘要表

**🔴 CHECKPOINT-0**：用户确认 `feapder_mode` 与诊断结论 → **停步**；用户回复「继续 Phase 1」后再进入 Phase 1。

## Phase 0.5：基础设施与反爬配套配置（必做，在 Phase 1 之后）

> 在 Phase 1 得出反爬等级后，按等级**有条件地**追问下列项。L1–L2 可跳过部分问题，但须显式标注「不需要」。

### A. 代理

| # | 问题 | 若「需要」则继续追问 |
|---|------|----------------------|
| P1 | 是否使用代理？ | — |
| P2 | 代理来源：自建 Redis 池 / SmartProxy / ScraperAPI / 其他 API / 用户提供列表 | 提供商名称 |
| P3 | 是否有现成 API 文档或网关 URL？ | **请用户提供文档链接或接口说明** |
| P4 | API Key / 网关账号密码？ | 用户本地配置，不落库到 skill |
| P5 | 代理类型：数据中心 / 住宅 / 轮换？预期 IP 数量？ | 影响并发与成本 |
| P6 | 失效策略：403/429 自动剔除？冷却回迁？ | 写入 proxy_middleware 设计 |

### B. 打码 / 验证码

| # | 问题 | 若「需要」则继续追问 |
|---|------|----------------------|
| C1 | 是否接入打码平台？（Phase 1 判定 L4+ 验证码时默认追问） | — |
| C2 | 验证码类型：滑块 / 点选 / 图片字符 / AWS WAF / 其他 | 决定转交 captcha-slide-reverse 或通用 OCR |
| C3 | 打码平台：2Captcha / CapSolver / 超级鹰 / 若快 / 其他 | 平台名称 |
| C4 | 是否有平台 API 文档？ | **请用户提供文档 URL 或 PDF** |
| C5 | API Key？ | 用户本地配置 |
| C6 | 预算与超时：单次 solve 超时秒数、失败重试次数 | 写入 middleware 设计 |

### C. 账号池

| # | 问题 | 若「需要」则继续追问 |
|---|------|----------------------|
| A1 | 是否需要多账号轮换？ | — |
| A2 | 账号来源：手动 Cookie / 自动登录 / Playwright 取态 | 转交 web-protocol-recovery |
| A3 | 每账号并发上限、封禁判定规则 | account_pool 设计 |

### D. 浏览器混合（L4+）

| # | 问题 |
|---|------|
| B1 | 是否启用 Playwright/Camoufox 取态？ |
| B2 | 取态频率：每 N 请求 / 定时 / 仅 bootstrap 一次 |
| B3 | 是否已有浏览器 profile 或 Cookie 导出文件 |

### E. 存储与中间件

| # | 问题 | 若「需要」则继续追问 |
|---|------|----------------------|
| S1 | Redis：地址、端口、DB、密码、是否集群/哨兵 | 连接串 |
| S2 | MySQL：地址、库名、表名前缀、是否已有表结构 | schema 或 DDL |
| S3 | MongoDB：地址、库名、集合名 | 连接串 |
| S4 | 结构化字段 vs 原始 HTML 的分工是否按默认（MySQL 结构化 + Mongo 原始） | 自定义则列出字段 |

### F. 调度与运维

| # | 问题 |
|---|------|
| O1 | 定时策略：Cron 表达式 / 外部调度系统 |
| O2 | 监控告警：是否需要成功率、403 率、队列积压告警 |
| O3 | 日志与数据保留周期 |

### Phase 0.5 输出

- 《基础设施配置摘要》表格（每项：是否需要 / 方案 / 待用户提供材料）
- `config/settings.py` **拟填字段清单**（不含真实密钥）

**🔴 CHECKPOINT-0.5**：用户确认基础设施方案与材料齐全度 → **停步**；用户回复「继续 Phase 2」后再进入 Phase 2（策略，非架构）。

## 缺材料时的行为

| 情况 | Agent 行为 |
|------|------------|
| 用户说「需要代理」但未给 API | 在架构中预留 Hook + 列出待填配置项；**不编造 Key** |
| 用户说「需要打码」但未给文档 | 追问文档；可给出该平台的公开文档搜索建议，**但不写具体集成代码** |
| 用户说「先用默认本地 Redis」 | 可写入 localhost 占位，并在 CHECKPOINT 明确标注「开发默认值」 |
| 用户要求「一次性全做」 | 仍先输出 Phase 0–4 文档；Phase 5 需单独确认 |

## 确认话术模板

Phase 0 结束（**本条回复必须在此结束，不得附带 Phase 1+**）：

```markdown
## 🔴 CHECKPOINT-0
- 已接收：（简要表格）
- 仍缺项：（Q1–Q4 及按需 Q5–Q9）
- Feapder 模式建议（tentative）：**{mode}** — 待确认

请确认以上摘要与模式建议。确认请回复「继续 Phase 1」；需修改请直接说明。
```

Phase 0.5 结束：

```markdown
## 🔴 CHECKPOINT-0.5
- 代理 / 打码 / 账号 / Redis / 存储：（各一行）
- 待您补充的材料：（列表）

确认请回复「继续 Phase 2」。
```

Phase 2 / 3 / 4 话术见 [checkpoint-protocol.md](checkpoint-protocol.md)。

Phase 4 结束：

```markdown
## 🔴 CHECKPOINT-4
（风险 Top + 合规标注）

确认请回复「开始落地脚手架」后才会创建项目文件。
```
